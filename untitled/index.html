    Decode a base32 string that may be missing '=' padding and may be lowercase.

    # Pad to a multiple of 8 chars as required by standard base32
    pad_len = (-len(s)) % 8

> Overview

  - Purpose: Adds SQLite file I/O utilities (read/write files, list directories) by loading a native SQLite extension into the mattn/go-sqlite3 driver.
  - Key point: It does not automatically “save files into SQLite”. It exposes SQL functions you can use to read files into the DB or write blobs from the DB out to the filesystem.

  Components

  - ext.go: Registers driver sqlite3-ext and loads shared library libgo-sqlite3-ext.so via LoadExtension("sqlite3_fileio_init").
  - src/fileio.c: SQLite extension implementing readfile(), writefile(), and the fsdir virtual table. Based on SQLite’s “fileio” extension.
  - CMakeLists.txt: Builds go-sqlite3-ext shared library (install name becomes libgo-sqlite3-ext.so).
  - ext_test.go: Smoke test that sql.Open("sqlite3-ext", ":memory:") loads the extension.

  What the extension provides

  - readfile(path): Returns the file at path as a BLOB (or NULL if unreadable).
  - writefile(path, data[, mode[, mtime]]): Writes data to filesystem path. Optionally sets POSIX mode and modification time.
  - fsdir(path[, dir]): Table-valued function listing files under path (columns: name, mode, mtime, data). For regular files, data is the file content blob; for symlinks, link text; for dirs, NULL.

  Does it save files to SQLite?

  - Not by itself. The extension allows you to move data between the filesystem and SQLite via SQL, but it doesn’t auto-ingest or auto-persist files.
  - How to save into SQLite: Use readfile() in an INSERT/UPDATE to store file bytes in your tables.
      - Example: INSERT INTO files(name, data) VALUES ('foo.png', readfile('/path/foo.png'));
  - How to write from SQLite to disk: Use writefile() with a blob selected from your DB.
      - Example: SELECT writefile('/out/foo.png', data) FROM files WHERE name='foo.png';

  Usage examples

  - Import a directory tree into a table:
      - INSERT INTO files(name, mode, mtime, data) SELECT name, mode, mtime, data FROM fsdir('/some/dir');
  - Export from a table back to the filesystem:
      - SELECT writefile('/restore/' || name, data, mode, mtime) FROM files;
  - Read a single file into a query result:
      - SELECT readfile('/etc/hosts');

  Build and use

  - Build and install the shared library:
      - mkdir build && cd build && cmake .. && make && make install
  - Open with the driver that loads the extension:
      - Go: db, _ := sql.Open("sqlite3-ext", "file:app.db?_busy_timeout=5000")
  - Ensure the loader can find libgo-sqlite3-ext.so (e.g., installed to a standard lib path or set LD_LIBRARY_PATH).

  Notes

  - readfile, writefile are created with SQLITE_DIRECTONLY for safety (callable in top-level SQL, not from triggers/views).
  - Using writefile() writes to the OS filesystem, not into the database.
  - If the library isn’t found, the driver returns ErrNotFound during connect.

> Short answer: yes. There are clear RCE paths if an attacker can write files in specific places, combined with functions already present in your app.

  Direct Plugin Loading

  - Code: utils.LoadPlugins(server.Options.Paths.Plugins) in main.go, which calls plugin.Open and invokes Init().Process() for every .so in that directory (utils/utils.go).
  - RCE Combo: Any ability to write a .so file into Options.Paths.Plugins leads to immediate arbitrary code execution on the next startup (or restart) because the plugin’s Process() runs in-process.
  - Variants:
      - If an attacker can also change config.yaml to repoint Paths.Plugins to a writable location, that plus a restart yields RCE.
      - If an attacker can trigger a restart (e.g., by inducing a controlled panic), newly dropped plugins will be loaded next run.

  SQL From Templates → SQLite Extensions

  - Code: Templates are parsed with html/template, and when data == nil you pass the entire *Server into templates (handlers.go:renderPage). That exposes .Database (*sql.DB) and .SQL methods to templates.
  - Result: A malicious template can call DB APIs, e.g., {{ .Database.Exec "SQL..." }} → arbitrary SQL execution.
  - RCE Combo: With SQLite “ext” build (github.com/MoshZillaRadio/go-sqlite3-ext), if loadable extensions are enabled, arbitrary SQL can run:
      - SELECT load_extension('/path/to/evil.so'); → loads native code into the process → RCE.
      - If file I/O extension functions (e.g., writefile()) are available/loaded, SQL alone can write arbitrary files; combined with plugin loading on restart, that’s also RCE.
  - Note: Whether load_extension is enabled depends on how the driver is built/configured at runtime. The “ext” fork strongly suggests extension support; if it’s enabled, this path is viable.

  Other File-Write Adjacent Surfaces

  - mount.DumpFile: If an attacker can alter config to set DumpFile to a chosen path, then connect as a source, they can write arbitrary bytes to that file. Combined with the plugin loader (point Paths.Plugins or
  drop into plugins dir), this can yield RCE on restart.
  - Static pages (404.html, 500.html) and templates: Writable by an attacker leads to defacement/XSS, but not by themselves to server-side code execution without the DB/extension/plugin vectors above.

  Why these are realistic

  - Templates are reparsed from disk each request (template.ParseFiles), so template compromise is “live”.
  - Exposing *Server to templates gives them direct reach to the DB and other exported methods, turning “template edit” into “arbitrary SQL”.
  - The plugin loader executes any .so it finds, by design.

  Mitigations (targeted)

  - Pass minimal view models to templates; never pass *Server, *sql.DB, or anything with dangerous exported methods.
  - Remove or gate utils.LoadPlugins in production, or run with a plugins directory that is not writable by the service or attackers.
  - Ensure SQLite loadable extensions are disabled; use the standard driver or explicitly disallow load_extension.
  - Protect template and config directories with strict permissions, or embed templates using Go embed.

  - GET /admin/metadata?mode=updinfo&mount=/Example&song=<very-long-string>
  - Requires a valid Authorization: Basic <base64(user:pass)> header.

  Why It Panics

  - In mosh/mount.go::meta the code computes metaSize as a single byte:
      - metaSize = byte(ceil(len(mStr)/16))
      - mStr is "StreamTitle='<song>';", so len(mStr) = len(song) + 15.
  - If ceil(len(mStr)/16) == 256, the byte overflows to 0, so meta is allocated with length 1.
  - The code then writes len(mStr) bytes into meta[1:], causing a slice write past end → panic “index out
  of range”.

  How To Trigger

  - Make len(mStr) fall in 4081..4096, i.e., set len(song) to 4066..4081 bytes (ASCII to avoid encoding
  surprises).
  - Example (replace host/creds/mount as needed):
      - MOUNT=/Example
      - AUTH=$(printf 'admin:password' | base64)  # use real creds
      - SONG=$(head -c 4066 /dev/zero | tr '\0' 'A')
      - curl:
      - `curl -G "http://127.0.0.1:9080/admin/metadata" --data-urlencode "mode=updinfo" --data-urlencode
  "mount=$MOUNT" --data-urlencode "song=$SONG" -H "Authorization: Basic $AUTH"`

  Effect

  - The panic message contains “index out of range”, so the middleware calls os.Exit(1).
  - With the provided compose config (restart: always and a runner loop), the process restarts
  immediately.
